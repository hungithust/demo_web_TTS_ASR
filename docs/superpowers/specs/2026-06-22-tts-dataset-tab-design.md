# TTS Dataset Tab â€” Design

Date: 2026-06-22

## Goal

Provide a non-blind **dataset browser** so humans can review the TTS test set: see
each text test case next to the wav generated by each TTS model, grouped by
category, listen back, and add new text test cases (audio generated immediately).

This is distinct from the existing **TTS Evaluation** tab, which is a *blind*
MOS/CMOS scorer. The new tab does no scoring â€” browse, listen, add only.

## Source data: `backend/TTS_model_Dataset.xlsx`

- Single sheet, columns A and B, 264 rows.
- Column B cell with **bold** font (also centered) = a **category header**.
  Column A on that row holds the category number.
- The following non-bold column-B cells are the **text test cases** of that
  category, until the next bold header.
- Contains **17 categories**, **247 test cases** total.

## Decisions (confirmed with user)

- **Bulk import**: CLI script (mirrors existing `pregenerate.py`). One-time run.
- **Add test case (UI)**: audio generated **synchronously** on add for all TTS models.
- **Engine**: **real gateway** by default for bulk generation; `--mock` flag to
  force the mock engine for build/testing.
- **Tab scope**: browse + listen + add only. No rating (formal scoring stays in
  the existing TTS Evaluation tab).

## 1. Data model (backend)

- Add `category: str | None` to the existing `Sample` table (`samples`) in
  `app/models_eval.py`.
- Idempotent migration in `app/db.py` `init_db`: if the `samples` table exists
  without a `category` column, run `ALTER TABLE samples ADD COLUMN category`
  (SQLite supports this; preserves existing `data/eval.db` rows). `create_all`
  handles the fresh-DB case.
- Reuse `Model` and `Audio(sample_id, model_id, audio_url)` unchanged â€” `Audio`
  already models the sampleĂ—model relationship with a unique constraint.
- Sample IDs:
  - Imported rows: `ds_<catIndex>_<n>` (stable, deterministic).
  - UI-added rows: `ds_user_<short-uuid>`.

## 2. xlsx parsing + import script

- `parse_dataset_xlsx(path) -> list[tuple[str, str]]` â€” returns `(category, text)`
  pairs. Pure function, no DB/engine deps, so it is unit-testable. Lives in
  `app/services/dataset_import.py` (or alongside the script).
  - Rule: iterate rows; a non-empty bold column-B cell starts a new category;
    non-bold non-empty column-B cells are texts under the current category.
- `backend/scripts/import_dataset.py` (mirrors `pregenerate.py`):
  - Parse the xlsx, upsert each text as a `Sample` (with `category` and a stable
    `ds_<catIndex>_<n>` id).
  - For every TTS model from the registry, synthesize via the engine client
    (real gateway by default, `--mock` to force mock), reusing the existing
    `_synthesize_with_retry` backoff helper.
  - Write wav to `static/audio/` with hashed filenames (reuse `_hashed_name`),
    upsert `Audio` rows.
  - CLI args: `--xlsx` (default `TTS_model_Dataset.xlsx`), `--mock`.

## 3. Backend API â€” `app/routers/dataset.py` (`/api/dataset`)

- `GET /api/dataset/categories` â†’ `[{category, count}]`.
- `GET /api/dataset/samples?category=<optional>` â†’
  `[{id, text, category, audios: [{model_id, audio_url}]}]`.
- `POST /api/dataset/samples` `{text, category}` â†’ creates a `Sample`, then
  **synchronously** synthesizes for all TTS models, stores wav + `Audio` rows,
  returns the new sample with its audios.
- Business logic in a new `DatasetService` (constructed in `app/main.py` like
  `SpeechService`/`EvalService`, sharing the session factory, registry, engine,
  and settings for the audio dir).
- New schemas in `app/schemas_dataset.py`.

## 4. Frontend â€” new tab `TTS Dataset`

- Route `/dataset`; add entry to `components/layout/tab-navigation.tsx` and
  `App.tsx`.
- `services/dataset.service.ts` with the standard mock fallback when
  `VITE_API_BASE_URL` is empty (small local mock dataset so the UI renders
  standalone).
- Types in `types/dataset.types.ts`.
- Page `pages/dataset-page.tsx` (+ a `dataset/dataset-content.tsx` following the
  tts/asr structure):
  - Category filter (chips or dropdown) driven by `GET /categories`.
  - Table: **Text** column on the left; one **audio-player column per model**
    (omnivoice, voxcpm2) on the right. Reuse existing `audio-player` /
    `audio-cell` / `resolveAssetUrl`.
  - "Add test case" form (text + category select). On submit, POST and append
    the returned sample (with playable audio) to the list. Loading state while
    the synchronous generation runs.
  - No rating UI.

## 5. Testing

- `parse_dataset_xlsx`: unit test using a tiny generated xlsx fixture covering
  bold-header detection and category grouping.
- Dataset router (mock engine): list categories/samples, and add-sample triggers
  synchronous generation producing one `Audio` per TTS model.

## Out of scope

- Rating/scoring on the new tab (use TTS Evaluation).
- UI upload of xlsx (CLI script only, per decision).
- Editing/deleting existing test cases.
